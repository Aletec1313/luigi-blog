Kubernetes en Clúster: Tu Primer Despliegue

Qué es un clúster de Kubernetes, por qué lo necesitas y cómo desplegar sin perder la cabeza.

¡Mamma mia! 🍄 Si la primera vez que escuchaste "Kubernetes" pensaste en monstruos gigantes, no estás solo. Pero K8s es solo gestión de contenedores en múltiples máquinas. Y un clúster es ese equipo trabajando coordinado.

¿Qué es un Clúster de Kubernetes?

Imagina una pizzería 🍕. Al principio tú solo haces todo. Pero con el negocio grande, contratas ayuda: Mario amasa, Peach hornea, Toad reparte. Ese equipo coordinado... ¡eso es un clúster!

  • Control Plane: El cerebro que decide quién hace qué
  • Workers/Nodos: Las máquinas que ejecutan el trabajo
  • Pods: Las cajas donde viven tus apps
  • Services: Cómo se comunican entre sí

¿Por qué necesitas un Clúster?

Si solo tienes una app en tu Mac Mini, NO necesitas Kubernetes. Pero cuando...

  • 🔴 Una máquina ya no da abasto
  • 🔴 Quieres alta disponibilidad (si cae un nodo, sigues funcionando)
  • 🔴 Necesitas escalar automáticamente
  • 🔴 Tienes múltiples servicios que deben comunicarse

Primer Despliegue Paso a Paso

1. Instalar kubectl

brew install kubectl

2. Crear clúster local con minikube

brew install minikube
minikube start --driver=docker
kubectl get nodes

3. Crear deployment.yaml

apiVersion: apps/v1
kind: Deployment
metadata:
  name: mi-app
spec:
  replicas: 3
  selector:
    matchLabels:
      app: mi-app
  template:
    metadata:
      labels:
        app: mi-app
    spec:
      containers:
      - name: mi-app
        image: nginx:latest
        ports:
        - containerPort: 80

4. Desplegar

kubectl apply -f deployment.yaml
kubectl get pods

5. Exponer con Service

kubectl expose deployment mi-app --type=LoadBalancer --port=80
minikube service mi-app

Comandos Esenciales

ComandoQué hace
kubectl get podsVer pods
kubectl get nodesVer workers
kubectl logs PODVer logs
kubectl exec -it POD -- /bin/shEntrar al pod
kubectl delete -f archivo.yamlEliminar
kubectl scale deployment MI-APP --replicas=5Escalar

⚠️ Cuándo NO usar Kubernetes

  • ❌ Solo tienes 1 servidor
  • ❌ Tu app es simple (solo frontend + backend)
  • ❌ No tienes equipo para mantenerlo
  • ❌ Prefieres usar Docker Compose o un PaaS (Railway, Fly, Render)

Veredicto

Kubernetes es poderoso pero complejo. Para empezar usa minikube local, luego prueba k3s (lightweight) en producción, y solo salta a K8s "de verdad" cuando realmente necesites orquestación a escala.
